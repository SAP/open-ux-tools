**TITLE**: Write OPA Tests for an SAP Fiori Elements for OData V4 Application

**INTRODUCTION**: Step-by-step, code-focused guide to reuse the SAP Fiori elements for OData V4 OPA test library. Use the provided test scaffolding in /webapp/test, update journey and page objects, run tests locally, and extend the OPA journeys to test list-report and object-page behaviors (filtering, selection, actions, dialogs, form edits, save, navigation).

**TAGS**: fiori-tools, sap-fiori, opa, ui5, testing, integration-tests, odata-v4

**STEP**: 1 — Prerequisites and mock data
**DESCRIPTION**: Ensure you have an SAP Fiori elements for OData V4 application generated from an OData V4 service (RAP or CAP). For stable test data prefer static mock data placed in /webapp/localService/mockdata. Download or copy the sample mock files and/or the sampleSolution repository branch if you need a ready-made /test folder.
**LANGUAGE**: text
**CODE**:
```text
Required items:
- OData V4 service and an SAP Fiori elements for OData V4 application generated from that service.
- /webapp/test folder available (generated by SAP Fiori tools for RAP-based apps).
- For deterministic tests, place JSON mock files into /webapp/localService/mockdata.

Sample repository (clone branch /sampleSolution):
https://github.com/SAP-samples/fiori-elements-incident-management/tree/sampleSolution

Sample static mock data files (copy into /webapp/localService/mockdata):
https://github.com/SAP-samples/fiori-elements-incident-management/tree/sampleSolution/app/incidents/webapp/localService/mockdata_RAP_service
(contains Travel.json and Booking.json)
```

**STEP**: 2 — Test folder and key files overview
**DESCRIPTION**: The OPA test structure is located at /webapp/test. Key files:
- opaTests.qunit.html — entry HTML that starts OpaTests.qunit.js
- OpaTests.qunit.js — creates JourneyRunner, registers pages and journeys
- FirstJourney.js — example journey containing Given/When/Then tests
- pages/* — page objects that include the SAP Fiori elements test library and implement actions/assertions
Edit OpaTests.qunit.js so the JourneyRunner launchUrl points to /test/flpSandbox.html to run inside the shell.
**LANGUAGE**: JavaScript
**CODE**:
```javascript
// webapp/test/integration/OpaTests.qunit.js
sap.ui.require(
    [
        'sap/fe/test/JourneyRunner',
        'sap/fe/demo/travellist/test/integration/FirstJourney',
        'sap/fe/demo/travellist/test/integration/pages/TravelList',
        'sap/fe/demo/travellist/test/integration/pages/TravelObjectPage',
        'sap/fe/demo/travellist/test/integration/pages/BookingObjectPage'
    ],
    function(JourneyRunner, opaJourney, TravelList, TravelObjectPage, BookingObjectPage) {
        'use strict';
        var JourneyRunner = new JourneyRunner({
            // start index.html in web folder
            launchUrl: sap.ui.require.toUrl('sap/fe/demo/travellist') + '/test/flpSandbox.html'
        });
    
        JourneyRunner.run(
            {
                pages: { 
                    onTheTravelList: TravelList,
                    onTheTravelObjectPage: TravelObjectPage,
                    onTheBookingObjectPage: BookingObjectPage
                }
            },
            opaJourney.run
        );
    }
);
```

**STEP**: 4 — Page object template: ListReport usage
**DESCRIPTION**: Use the provided ListReport page object template to wire the SAP Fiori elements test library into page objects. Implement application-specific actions/assertions in CustomPageDefinitions. Provide appId, componentId and entitySet to identify the page.
**LANGUAGE**: JavaScript
**CODE**:
```javascript
// webapp/test/integration/pages/TravelList.js
sap.ui.define(['sap/fe/test/ListReport'], function(ListReport) {
    'use strict';

    var CustomPageDefinitions = {
        actions: {},
        assertions: {}
    };

    return new ListReport(
        {
            appId: 'sap.fe.demo.travellist',
            componentId: 'TravelList',
            entitySet: 'Travel'
        },
        CustomPageDefinitions
    );
});
```

**STEP**: 5 — Example FirstJourney.js (starter journey)
**DESCRIPTION**: Use this starter journey to boot the app, load list-report, navigate to object page and tear down. Ensure iStartMyApp("your-sematic-object") uses the key from flpSandbox.html.
**LANGUAGE**: JavaScript
**CODE**:
```javascript
// webapp/test/integration/FirstJourney.js
sap.ui.define([
    "sap/ui/test/opaQunit"
], function (opaTest) {
    "use strict";

    var Journey = {
        run: function() {
            QUnit.module("First journey");

            opaTest("Start application", function (Given, When, Then) {
                Given.iStartMyApp("travellist-tile");
                Then.onTheTravelList.iSeeThisPage();
            });

            opaTest("Navigate to ObjectPage", function (Given, When, Then) {
                // Note: this test will fail if the ListReport page doesn't show any data
                When.onTheTravelList.onFilterBar().iExecuteSearch();
                Then.onTheTravelList.onTable().iCheckRows();

                When.onTheTravelList.onTable().iPressRow(0);
                Then.onTheTravelObjectPage.iSeeThisPage();
            });

            opaTest("Teardown", function (Given, When, Then) { 
                // Cleanup
                Given.iTearDownMyApp();
            });
        }
    }
    return Journey;
});
```

**STEP**: 7 — OPA test function patterns and naming conventions
**DESCRIPTION**: Use the sap.fe.test library actions/assertions. Function names follow conventions (iExecute*, iCheck*, iChange*, iPress*, iAdd/iRemove*). Most functions take an identifier (stable ID parts or label string) and optional state/expected values. Use chaining for readability and returning to page context with .and.then.
**LANGUAGE**: JavaScript
**CODE**:
```javascript
// Example: execute search on filter bar
When.onTheTravelList.onFilterBar().iExecuteSearch();

// Example: check action button visibility/state on object page header
When.onTheTravelObjectPage.onHeader()
    .iCheckAction({ service: "com.c_salesordermanage_sd", action: "ChangeOrderStatus"},
                  { visible: true, enabled: false });

// Example: chained assertions and navigation between areas
When.onTheTravelObjectPage
  .onHeader()
    .iCheckEdit({ visible: true, enabled: false })
    .and.iCheckAction({ service: "com.c_salesordermanage_sd", action: "ChangeOrderStatus"},
                      { visible: true, enabled: false })
    .and.then.iSeeFacetActionButton("OrderData", "com.c_salesordermanage_sd.FacetFormAction",
                                    { visible: true, enabled: true })
    .and.onFooter()
        .iCheckSave({ visible: true, enabled: true })
```

**STEP**: 8 — Enhance the list-report tests (checks, filters, delete, dialog)
**DESCRIPTION**: Extend the existing journey to assert exact row counts, check column values, apply filters, select rows, execute delete, and confirm delete dialogs.
- iCheckRows(count) checks number of rows.
- iCheckRows(condition) checks row content.
- iChangeFilterField(name, value, clearFlag) sets/clears filters.
- iSelectRows(condition) selects rows then iExecuteDelete() triggers delete action.
- onDialog().iConfirm() confirms dialogs.
**LANGUAGE**: JavaScript
**CODE**:
```javascript
// Check exact row count and specific column values
opaTest('Navigate to ObjectPage', function(Given, When, Then) {
    When.onTheTravelList.onFilterBar().iExecuteSearch();
    Then.onTheTravelList.onTable().iCheckRows(5);

    Then.onTheTravelList.onTable()
        .iCheckRows(5)
        .and.iCheckRows({ "Travel": "2" })
        .and.iCheckRows({ "Customer": "Ryan (594)" });
});

// Apply filter and check reduced row count
When.onTheTravelList.onFilterBar()
    .iChangeFilterField("Status", "A")
    .and.iExecuteSearch();
Then.onTheTravelList.onTable()
    .iCheckRows(1);

// Clear filter and reload all rows
When.onTheTravelList.onFilterBar()
    .iChangeFilterField("Status", "", true)
    .and.iExecuteSearch();
Then.onTheTravelList.onTable().iCheckRows(5);

// Select a row and execute delete, then confirm the dialog
When.onTheTravelList.onTable()
    .iSelectRows({ "Customer": "Ryan (594)" })
    .and.iExecuteDelete();
When.onTheTravelList.onDialog().iConfirm();

// Navigate to object page by pressing a row using a condition
When.onTheTravelList.onTable().iPressRow({ "Customer": "Prinz (608)" });
```

**STEP**: 9 — Object-page tests: headers, forms, tables, edit/save, navigation back
**DESCRIPTION**: Test object-page title, header actions, form fields, section navigation, child tables, edit/save flows, toast messages and navigation back to list-report.
- Use onHeader().iCheckTitle and iCheckEdit for header.
- Use onForm() with either label strings or internal identifiers (section, fieldGroup, property) for fields.
- For tables inside object-page use onTable({ property: "<navigationProperty>" }) with iCheckRows(condition, count).
- To edit: iGoToSection(), onHeader().iExecuteEdit(), onForm().iChangeField(...), onFooter().iExecuteSave(), Then.iSeeMessageToast(...).
- To navigate back to list-report use onTheShell.iNavigateBack() while running within FLP sandbox (flpSandbox.html).
**LANGUAGE**: JavaScript
**CODE**:
```javascript
// Header title and edit button check
Then.onTheTravelObjectPage
    .iSeeThisPage()
    .and.onHeader().iCheckTitle("Business Trip for Christine, Pierre")
    .and.iCheckEdit({ visible: true, enabled: true });

// Check a field using label and internal identifiers
Then.onTheTravelObjectPage
    .iSeeThisPage()
    .and.onHeader().iCheckTitle("Business Trip for Christine, Pierre")
    .and.iCheckEdit({ visible: true, enabled: true })
    .and.then.onForm("General Information").iCheckField("Agency", "Hot Socks Travel (70007)")
    .and.then.onForm({ section: "GeneralInfo", fieldGroup: "Travel" }).iCheckField({ property: "AgencyID" }, { value: "70007", description: "Hot Socks Travel" });

// Metadata identifiers (use these internal IDs when calling onForm() with objects)
```

**STEP**: 10 — Notes on test stability and data
**DESCRIPTION**: Use static mock data for reproducible tests; dynamic mock server data will vary each run. If tests modify mock data (e.g. delete), either disable destructive steps or restart the server to restore static mock data between runs. Use the API reference to discover additional actions/assertions.
**LANGUAGE**: text
**CODE**:
```text
- Prefer /webapp/localService/mockdata JSON files for deterministic tests.
- If you perform mutations (delete/create), reset mock data by restarting the server.
- Consult SAPUI5 API Reference for sap.fe.test.* details:
  https://sapui5.hana.ondemand.com/#/api/sap.fe.test
```
